ES6 export as a Building Block

Publication date: 2022-05-08

What is a building block (brick) in modern ES2015+ applications? A variable, function, class, or module? My answer is export. Today we organize JavaScript code as ES6 modules and connect them via import:

import {export1, export2} from "module-name";

Each export is a minimal code object from which complex programs can be built.

Script Files

Code is typically placed in separate files - modules. An ES6 module can contain multiple classes, functions, and objects. The code executes on the first import, and its result is saved in the module's scope:

// cube.mjs
export default function cube(x) {
    return x * x * x;
}

To access an export, you need to know the file path:

// main.mjs
import cube from './cube.mjs';
const y = cube(2); // 8

Paths can be absolute or relative. Node.js also supports package name paths:

import cube from '@vendor/package/path/to/cube.mjs';

Important: relative paths must start with ./. Otherwise - error:

Relative references must start with either "/", "./", or "../".

First import

Module code executes only once - on the first import. Subsequent imports use already computed exports.

Module Scope

Example module with a hello function that changes internal state:

// sub.mjs
console.log(`es6-module is imported.`);
let from = 'module';

export function hello(to) {
    console.log(`Hello to ${to} from ${from}.`);
    from = to;
}

Even with multiple imports, the message will appear only once:

import {hello} from './sub.mjs';

hello('main');
hello('again');
es6-module is imported.
Hello to main from module.
Hello to again from main.

Async import

A module can perform async initialization and export the result later:

console.log(`es6-module is imported.`);
const init = new Promise((resolve) => {
    setTimeout(() => {
        function hello(to) {
            console.log(`Hello to ${to} from ${from}.`);
            from = to;
        }
        resolve(hello);
    }, 3000);
});
let from = 'module';

const hello = (await init);
console.log(`'hello' function is created.`);
export {hello}
import {hello} from './async.mjs';

hello('main');
hello('again');
es6-module is imported.
'hello' function is created.
Hello to main from module.
Hello to again from main.

Module Without Exports

A module can perform useful actions without exports:

function cleanUpFiles() {}

setInterval(() => {
    cleanUpFiles();
}, 60 * 1000);

Importing such a module still activates it:

import './cleaner.mjs';

Conclusion

Each export is a building block (brick) for creating applications (wall). I think in terms of Path_To_Module.exportName - both in documentation and code:

class Demo {
    constructor(spec) {
        const castInt = spec['TeqFw_Core_Shared_Util_Cast.castInt'];
        const castStr = spec['TeqFw_Core_Shared_Util_Cast.castStr'];
    }
}

This is an example of using the @teqfw/di dependency container, with which I build complex applications from simple modules.

I'm convinced: JavaScript needs namespaces like "serious" languages. I plan to continue this topic in future posts.
