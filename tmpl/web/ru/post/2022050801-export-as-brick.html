ES6 export как строительный блок

Дата публикации: 2022-05-08

Что такое строительный блок (brick) в современных приложениях на ES2015+? Переменная, функция, класс, модуль? Мой ответ — export. Сегодня мы организуем JavaScript-код в виде модулей ES6 и связываем их через import:

import {export1, export2} from "module-name";

Каждый export — это минимальный объект кода, из которого можно собрать сложную программу.

Файлы скриптов

Обычно код размещают в отдельных файлах — модулях. ES6-модуль может содержать несколько классов, функций и объектов. Код выполняется при первом import, и его результат сохраняется в области видимости модуля (scope):

// cube.mjs
export default function cube(x) {
    return x * x * x;
}

Чтобы получить доступ к export, нужно знать путь к файлу:

// main.mjs
import cube from './cube.mjs';
const y = cube(2); // 8

Пути могут быть абсолютными или относительными. Для Node.js поддерживаются и пути с именем пакета:

import cube from '@vendor/package/path/to/cube.mjs';

Важно: относительные пути должны начинаться с ./. Иначе — ошибка:

Relative references must start with either "/", "./", or "../".

Первый import

Код модуля выполняется только один раз — при первом import. Повторный импорт использует уже вычисленные export.

Область модуля

Пример модуля с функцией hello, которая меняет внутреннее состояние:

// sub.mjs
console.log(`es6-module is imported.`);
let from = 'module';

export function hello(to) {
    console.log(`Hello to ${to} from ${from}.`);
    from = to;
}

Даже при многократном импорте сообщение будет выведено один раз:

import {hello} from './sub.mjs';

hello('main');
hello('again');
es6-module is imported.
Hello to main from module.
Hello to again from main.

Асинхронный import

Модуль может выполнять асинхронную инициализацию и экспортировать результат позже:

console.log(`es6-module is imported.`);
const init = new Promise((resolve) => {
    setTimeout(() => {
        function hello(to) {
            console.log(`Hello to ${to} from ${from}.`);
            from = to;
        }
        resolve(hello);
    }, 3000);
});
let from = 'module';

const hello = (await init);
console.log(`'hello' function is created.`);
export {hello}
import {hello} from './async.mjs';

hello('main');
hello('again');
es6-module is imported.
'hello' function is created.
Hello to main from module.
Hello to again from main.

Модуль без экспортов

Модуль может выполнять полезные действия без export:

function cleanUpFiles() {}

setInterval(() => {
    cleanUpFiles();
}, 60 * 1000);

Импорт такого модуля по-прежнему активирует его:

import './cleaner.mjs';

Итог

Каждый export — это строительный блок (brick) для создания приложения (wall). Я мыслю в терминах Path_To_Module.exportName — как в документации, так и в коде:

class Demo {
    constructor(spec) {
        const castInt = spec['TeqFw_Core_Shared_Util_Cast.castInt'];
        const castStr = spec['TeqFw_Core_Shared_Util_Cast.castStr'];
    }
}

Это пример использования контейнера зависимостей @teqfw/di, с которым я собираю сложные приложения из простых модулей.

Убеждён: JavaScript нужны пространства имён, как в «серьёзных» языках. Планирую продолжить тему в следующих постах.
