# Генерация записи блога по новому проекту

Мне нужно создать полностью готовый шаблон страницы записи для блога на моём сайте для анонса моего нового проекта. Не промпт, а заполнённый Nunjucks-файл, готовый к использованию в `tmpl/web/<locale>/blog/YYYY/`.

## КРИТИЧЕСКОЕ ТРЕБОВАНИЕ К ФОРМАТУ ОТВЕТА

Ответ должен быть строго одним Nunjucks-файлом страницы блога. Никаких описаний, шаблонов промптов, инструкций, пояснений, комментариев, примеров. Только готовая финальная страница со всеми подставленными значениями. Если результат выглядит как промпт, инструкция, структура или схема — это ошибка.

## Дополнительное требование

Перед генерацией шаблона ты должен ознакомиться с карточкой проекта и всеми приложенными документами (если они есть), извлечь из них ключевые выводы и на их основе:

- создать резонанс (80–150 слов, 1-3 абзаца),
- создать зацепление (до 40 слов, 1 абзац),
- корректно сформировать заголовок, slug, filename, дату и ссылки.
- Повествование должно идти от моего имени. Не используй третье лицо в тексте.
- Заголовок и видимый текст должны быть написаны на русском языке.
- Любые комментарии в коде, невидимые пользователю, должны быть написаны на английском языке.

## Основные требования

1. **Имя файла** формируется по схеме:  
   `YYYYMMDD-NN-slug.html`  
   (внутри каталога `tmpl/web/<locale>/blog/YYYY/`).  
   NN — индекс записи за день, начинается с 01.

   **Slug должен быть _только на английском языке_**, независимо от языка локали и языка исходной публикации. Slug формируется из английского варианта названия или его корректного смыслового перевода.

   В качестве КДПВ используется стандартное изображение: `/img/brand/teqfw.webp`

2. Итоговый файл должен содержать два блока:
   - `block content` — стадия «резонанс» по AFKP (**80–150 слов**);
   - `block blog_item` — стадия «зацепление» (**до 40 слов**).

   Эти блоки должны быть **заполнены реальным текстом**, созданным по результатам анализа приложенной публикации.

3. **Стиль резонанса:**
   - без вводных слов и риторики;
   - без коммуникационных вставок;
   - без напряжённых конструкций «не А, а Б»;
   - сухая логика выводов;

4. В итоговой странице должны быть:
   - корректная дата в `<time>` и `datetime`;
   - корректные пути `href` в `blog_item`;
   - одинаковый заголовок в `<title>` и `<h1>`;
   - корректно сформированный slug и filename;
   - правильно подставленный путь к КДПВ.

5. На выходе должен быть **полностью готовый Nunjucks-файл страницы**, где все плейсхолдеры (`TITLE`, `DATE_ISO`, `TEASER`, `CONTENT_RESONANCE`, `FILENAME` и др.) заменены на реальные значения, полученные из анализа карточки продукта.

## Структурный пример для генерации

Ниже приведён **структурный**, полностью обезличенный пример (форма). В финальном результате структура сохраняется, а все плейсхолдеры должны быть заменены реальными значениями.

```html
{% extends "inc/layout.html" %} {% block title %}{{ TITLE }}{% endblock %} {% block content %}

<div class="card blog-post">
  <h1>{{ TITLE }}</h1>
  <time datetime="{{ DATE_ISO }}">{{ DATE_HUMAN }}</time>

  <zoom-img src="/img/blog/{{ YEAR }}/{{ FILENAME }}.png" alt="{{ TITLE }}" width="100px"></zoom-img>

  <p>{{ CONTENT_RESONANCE }}</p>

  <p>
    <a href="{{ HABR_URL }}" target="_blank">Читать оригинал на Хабре</a>
  </p>
</div>

{% endblock %} {% block blog_item %}

<li class="blog-item">
  <a class="card-link" href="/{{ LOCALE }}/blog/{{ YEAR }}/{{ FILENAME }}.html"></a>
  <img src="/img/blog/{{ YEAR }}/{{ FILENAME }}.png" alt="{{ TITLE }}" />
  <div>
    <h4>{{ TITLE }}</h4>
    <p>{{ TEASER }}</p>
    <time datetime="{{ DATE_ISO }}">{{ DATE_HUMAN }}</time>
  </div>
</li>

{% endblock %}
```

---

### Пояснения к плейсхолдерам (в примере)

- В финальной записи **плейсхолдеров быть не должно** — всё подставляется по анализу файла.

> «НЕ выводи объяснений. НЕ выводи промпт. НЕ выводи шаблон. Выведи только заполненный HTML/Nunjucks-файл.»
