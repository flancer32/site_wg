# Демонстрационные приложения по уровням

Ниже — четыре самодостаточных демо-приложения: по одному на каждый уровень продукта. Каждое приложение строго укладывается в рамки своего уровня и оставляет после себя реиспользуемые артефакты.

Вот кодовые названия для уровней: Solo / Combo / Concierge / Office.

---

## L1 — «Пресейл-ассистент продукта» (Custom GPT внутри ChatGPT)

**Назначение.** Показать, как без кода и интеграций квалифицировать лида, ответить на FAQ и выдать черновик оффера.

**Что видит заказчик.**

- Ссылка на Custom GPT со starter-подсказками и загруженным «пакетом знаний».

**Поведение.**

- 3–5 уточняющих вопросов (ниша, цель, материалы).
- Краткое объяснение уровней 1–4.
- Сбор брифа и генерация текста письма/оффера.

**Что подготовить.**

- `gpt/system.md` — правила голоса и шаги диалога.
- `gpt/starters.md` — 8–12 starter prompts.
- `gpt/knowledge/` — `overview.md`, `FAQ.md`, `pricing.md`, мини-кейсы.

**Результат.**

- Рабочий Custom GPT, готовый к показам и пилотам. **Кода нет.**

## L2 — «Intake & Knowledge Proxy» (Actions + внешний API)

**Назначение.** Показать доступ к данным через безопасный сервер-прокси по OpenAPI.

**Что видит заказчик.**

- Диалог с Actions: «Оставить заявку», «Поиск по базе знаний», «Черновая оценка».

**Поведение.**

- `/intake/create` — запись заявки (имя, задача, контакт).
- `/kb/search` — поиск по материалам и возврат выдержек.
- `/estimate/basic` — ориентир бюджета по простым правилам.

**Что подготовить (код).**

- `server/` (Node.js, Express): реализация трёх эндпойнтов.
- `api/openapi.yaml` + `tests/postman_collection.json`.
- Безопасность: сервисный API-ключ, rate-limit, логирование.
- Источник данных (любой один): FS/S3/PostgreSQL с адаптером.

**Результат.**

- Демо интеграции через Actions с минимальной безопасностью и повторно используемым прокси.

## L3 — «Client Hub: действия от имени пользователя» (OAuth)

**Назначение.** Показать персональные действия «от имени пользователя» с правами и аудитом.

**Что видит заказчик.**

- Веб-интерфейс: «Подключить сервис» (например, Google Drive/GitHub) → «Создать черновик SOW/Issue».

**Поведение.**

- OAuth авторизация; получение/хранение токенов (шифрование, срок, отзыв).
- Создание артефакта «от имени пользователя» и возврат ссылки.
- Аудит: кто/что/когда/результат.

**Что подготовить (код).**

- `auth/` — Authorization Code Flow (server-side).
- `storage/` — хранилище токенов (AES), таблица аудита.
- `docs/templating/` — генератор SOW с подстановкой полей.
- `web/` — мини-UI: подключение, создание черновика, просмотр лога.

**Результат.**

- Самодостаточное демо on-behalf сценариев; готовые модули OAuth, аудит, шаблоны документов.

## L4 — «Pre-Sales PWA» (офлайн, push, мобильный сценарий)

**Назначение.** Показать установимое мобильное приложение пресейла: входящие заявки, быстрые ответы, офлайн-черновики, уведомления.

**Что видит заказчик.**

- PWA с установкой на телефон; экраны: «Список заявок», «Карточка», «Ответ/черновик», «Настройки».

**Поведение.**

- Просмотр/фильтр заявок (REST).
- Оффлайн-черновики (IndexedDB) + синхронизация при сети.
- Web Push уведомления о новых заявках.

**Что подготовить (код).**

- `pwa/manifest.json`, `pwa/service-worker.js` (cache-first для shell, network-first для данных).
- `app/` (Vue + CSS): список, карточка, форма ответа, индикатор синхронизации.
- `server/notifications/` — эндпойнты пуш-подписок и доставки уведомлений.

**Результат.**

- Демонстрация мобильного процесса пресейла; реиспользуемые блоки PWA/офлайн/пуш.
